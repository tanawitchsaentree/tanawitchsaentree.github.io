## The problem

The people who understood how documents should be classified — the business configurators — had no way to act on that knowledge directly. Every adjustment to how the AI behaved required an engineering ticket, a deployment window, and days of waiting.

When the AI misclassified something, the debugging process was: file a ticket, wait for an engineer to read the prompt out loud to you, decide whether to change it, file another ticket.

The feedback loop was so long that most configurators had stopped trying to improve the system at all. They'd accepted the AI's behavior as fixed — even when they knew it was wrong.

A prompt is a policy document. It should have a draft, a review, and a publish — not a merge request and a deployment pipeline. The interface had to make that clear without making the engineering team feel bypassed.

## Trade-off ที่ยังอยู่

**No A/B testing between configurators.** Right now prompt changes are sequential — you publish version 3, see what happens, publish version 4. There's no way for two configurators to run competing prompt strategies against the same document type and compare results. That's a real gap when different teams have different opinions about classification behavior.

**Approval workflow is synchronous.** A configurator drafts a prompt, it goes to a reviewer, reviewer approves or rejects. If the reviewer is unavailable, the change waits. For time-sensitive adjustments — say, a new document type appearing in volume — that lag matters. We discussed async approval paths but didn't build them.
