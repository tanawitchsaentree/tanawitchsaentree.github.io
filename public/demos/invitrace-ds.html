Invitrace Health · 2024–2025 · Federated design system

One system. Every hospital's UI, rebuiltrethemed.

We license one clinical platform to District hospitals, Teaching hospitals, and Specialty clinics — same engine and workflows — but each needs its own brand, data density, and accessibility floor. Hand-building that UI per client was the bottleneck, so I made the interface a theme: colour, density, and a11y, all the way down to the atom.

The cost

Selling the platform was a day. Re-skinning it was a month.

Every hospital got a hand-built interface from zero — and in clinical software the drift wasn't just ugly, it was risky: inconsistent ward fields, contrast that failed older clinicians, an "Admit" button that meant three different things across three codebases.

The bet & the payoff

We could win the deal in a day. We couldn't ship in under a month.

Every signed hospital meant rebuilding the same interface by hand — roughly a designer-month per deal, with accessibility audits failing late in review. The cost wasn't taste; it was velocity and regulatory risk, and it scaled with every new client.

The system & the craft

Three hospitals. Three hand-drawn versions of the same component.

With no shared token, one "Admit patient" button drifted into a dozen variants — different padding, radius, contrast, label. Drift like this isn't a taste failure; it's a missing constraint. The system never actually owned the component.

The machine & the guarantees

One codebase. Three forks. Zero shared tokens.

Each hospital install was effectively its own codebase, so a single contrast fix had to be written three times — and could silently regress in any of them. Three UIs is three places a patient-safety bug hides, with no token layer to catch it.

Making it unbreakable

In healthcare, a broken component isn't a style bug.

A detached card that skips the contrast token can fail a patient who can't read it. So the kit is built so the three habits that rot a system simply don't pay off.

The bet & the payoffMaking it unbreakable

I designed for the people who'd misuse the system.

Governance is the unglamorous work that decides whether a design system survives a real team under deadline. Below is how I made the wrong move stop paying off — change the master token and watch what stays in line.

The system & the craftMaking it unbreakable

Detach, copy, invent — the three ways a system rots.

Every Figma library decays the same way. The fix isn't a rules doc; it's making the correct path the easiest one so drift never starts. Poke the stage below to see each failure — and why the system resists it.

The machine & the guaranteesMaking it unbreakable

Bound instances propagate; forks go stale.

An instance wired to the master updates with it; a detached or copied one freezes. Same reason you don't fork a shared component in code — and the same fix: one source, enforced. Change the master below and watch propagation.

Master token · source of truth
color/primary → #6d28d9

Detach

Detaching to tweak one value also drops the a11y token. Fix: every real need is a prop — including contrast mode — so there's nothing to detach for.

Copy-paste

A pasted copy looks right today and silently rots tomorrow. Fix: the kit is the only on-ramp — inserting is faster than rebuilding.

Invent

A hospital's "special" one-off becomes the tenth dosage field. Fix: new patterns enter by contribution and review — forking outside is the friction.

Every component ships to Storybook with its states and an a11y check baked in — so clinical teams', engineering, and QA all read the same source of truth, not a screenshot in a thread.
